我用多 Agent 开发流程做了好几个月了。一个模型写设计，一个写代码，一个做审查，我批结果。大部分时候挺顺的。

但最近有三个问题穿过了整个流水线，走到线上才被发现。它们有一种共同的模式——这个模式我之前一直说不清楚，直到三个摆在一起才看明白。

这三个 bug 本身不算什么。根因都不复杂，一眼看完就知道是怎么回事。有意思的不是 bug 本身，是它们暴露出来的那层结构：这类失败到底是什么，又该在什么检查点提前拦住它。

---

## 案例一：一次也没有触发过的重试回路

一个 ETL 任务在对接第三方的 ERP API，每 5 到 15 分钟拉一次增量数据。代码里有标准的重试逻辑：三次 attempts，指数退避，失败时调 `resetLogin()` 重新认证再试下一轮。已经稳定跑了好几周。

某天下午，所有同步任务开始报同一个错：`COPY from stdin failed: expected N values, got 1`。Pipeline 报错，重试，再报错，再重试，循环了几个小时，没人发现。

重试回路在代码里。`resetLogin()` 在代码里。一个也没触发过。

查了一阵才搞清楚。ERP 的 session 大概 8.5 小时过期。过期之后，API 返回的不是 401，不是 403，不是任何 HTTP 错误码，而是 HTTP 200。Body 解码出来是这样的：

```json
[["error message"]]
```

JSON 结构完全合法，一行一列。Go 的解码器没理由报错。重试条件判断的是 `if err != nil`，取到了 nil，就把这一行当成正常结果集往下走了。

修复方案不是改进重试逻辑。是在后台起了一个 goroutine，每 4 小时主动调一次 login——把 session 续上，不给它过期的机会。那条为了应对这种场景专门存在的重试回路，从上线第一天就是死的。

## 案例二：从来没有存在过的坐标

一个桌面工具里的代码生成器，负责给嵌入式 MCU 产出 C 源码。用户画好 UI 界面，点生成代码，工具产出一堆文件：头文件里有宏定义，有事件分发函数，有资源索引，指向外挂 Flash 里的图片地址。

开发者把这些生成代码编进固件，烧进芯片，屏幕亮了。动态文字控件全挤在 (0, 0)。

代码编译没 warning。链接器找到了所有符号。没有任何函数报错。一切看起来都完整。

但缺了一个文件——控件实例表。这个文件记录每个控件的 ID 和它的 x、y、宽、高，MCU 运行时靠它定位控件坐标。头文件里声明了控件结构体，结构体里有 x、y 字段。资源 manifest 里有坐标数据。设计文档里写清了控件表的格式。代码生成器就是没把它写出来。那个生成方法，从来没有人调用过。

审查过的。审查的人确认了头文件宏指向了正确的资源地址，事件路由逻辑对，页面定义一致。这些都是对的。只是没有一个人问：该生成的文件，都生成了吗？

## 案例三：看指令，别看注释

我在给一颗新的 MCU 集成 SDK。厂商给了一个 zip 包，目录结构都正常：启动文件、链接脚本、芯片型号对应、SPI、DMA、GPIO 驱动全在。

`startup.S` 文件头注释写着：

```asm
; GCC for CSKY Embedded Processors
```

CSKY 是另一种嵌入式 CPU 架构，不是这颗芯片用的。Datasheet 里写得很清楚，这是 RISC-V。但 SDK 目录名标的就是这颗芯片，子目录也对，启动文件就待在它应该在的地方。注释说的是一种架构，目录名说的是另一种。哪个对的？

两个事实把疑案拆开了。

第一，厂商第一次给的 SDK 其实是同系列另一颗芯片的——那颗确实是 CSKY。目录名是误导。第二，正确的 SDK 到了之后，它的 `startup.S` 头注释还是写的 CSKY。因为厂商从第三方工具链模板复制过来，就没改过。

实际指令不会说谎：

```asm
csrw    mtvec, a0
la      sp, g_top_irqstack
jal     main
```

最简洁的验证就是一行 grep：`grep -c "csrw\|mret" startup.S`，返回 3。RISC-V，没错。注释是化石。

## 不是能力问题

三个 Agent——设计的、实现的、审查的——各自做了一件单独来看完全正确的事。设计写了与任务一致的 spec。实现写出了与 spec 一致的代码。审查确认了一致性。每一步验证都通过了。系统是一致的。是**一致地错了**。

这不是能力问题。这是边界问题。

Agent 做了你要求它做的事，在你给它的上下文范围内。没有人告诉它上下文在哪结束。

案例一，上下文是"检查 HTTP 调用有没有返回错误"。没有人说"再检查一下成功响应里到底有没有数据"。

案例二，上下文是"检查生成的文件内容对不对"。没有人说"再检查一下该生成的文件是不是都存在"。

案例三，上下文是"信任 SDK 目录结构和文档"。没有人说"用实际的指令流验证架构声明"。

每次都一样。Agent 在它理解的范围内做了完整的验证。范围本身是错的。而那个缺口——Agent 检查的东西和实际应该为真的东西之间——是 AI 辅助流程从结构上就看不到的故障模式。

这不是 Agent 的 bug。这是 workflow 里缺了一层。Agent 会验证你让它验证的东西。它不会自己去发现新的验证类别。那个发现，最终仍是工程。

---

## Checklist 上该放什么

这三条，我现在写进了自己的工作流。

**一、语义验证——检查"成功"是什么意思，而不只是检查它返回了成功。**

API 调用、函数返回、系统调用——每一条都有成功路径和失败路径。但有些系统把失败藏在成功里面。每次和外部交互之后，问一句：返回的数据看起来像数据，还是像伪装成数据的错误？检查行数。检查结构。搜一下错误关键词。如果调用方请求的是很多列却只收到一行，不管你取的 `err` 是什么，那个数据已经可疑了。

**二、产物完整性——验证该有的文件都在，而不只是已有的文件内容都对。**

代码生成产出一堆文件。有的躺在输出目录里看得见。有的应该在，但没有。接受生成结果之前，把预期输出文件列出来，逐一确认存在且非空。不要用"生成出来的文件看起来没问题"替代"所有该生成的文件都在"。

**三、信息源验证——任何可以用机器检验的技术声明，就用机器检验。**

注释会说谎。文件名会说谎。目录结构会说谎。一个 `.S` 文件的 CPU 架构不是头注释说了算，是汇编器会输出的指令集说了算。一个二进制文件的格式不是 README 说了算，是特定偏移位置的字节值说了算。在 Agent 流程靠任何一个技术假设往下走之前，用一条能直接读取那个 artifact 的命令验证它。不要把未经检验的声明传给下一步。

---

三条都不大。每一条都小到可以直接写进任务定义或者审查阶段，不用动整个流程。合在一起，刚好覆盖了我反复遇见的三类失败：成功响应里藏着的错误、代码生成器的缺失输出、以及工具链制品里循环再生的虚假声明。

Agent 不会问自己"我还应该验证什么"。那个问题——"在一切我能检查的都正确的情况下，还有什么可能也是错的"——不是一个 Agent 能从它自己的上下文里推导出来的。它只能从外面来。必须被人设计。

你搭多 Agent 工作流的时候，不光是在设计一条生成代码的流水线。你是在设计每个 Agent 会相信什么、会把什么当真。如果你不在那些信念的边界上设置验证点，没有人会替你放。
