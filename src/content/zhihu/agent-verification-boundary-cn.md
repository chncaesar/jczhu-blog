我已经用多 Agent 的方式开发了好几个月。大致流程是这样：一个模型出设计文档，另一个照着写代码，第三个做审查。我在最后一关批结果。

大多数时候挺顺的。出活快，质量也不差。

但最近有三个问题，全穿过了这条流水线。设计过了，实现过了，审查过了，我的审批也过了。代码推上去，线上崩了。

三个 bug 的根因都简单，排起来花不了多少时间。有意思的不是 bug 本身——是它们为什么能穿透。穿透了之后再回头看，你会发现这三个东西是同一种原因。

---

## 一

有个 ETL 服务在对接一套 ERP 系统。每 5 到 15 分钟拉一次增量数据，session 登录一次后带 cookie 用。代码里写好了重试：三次 attempts，指数退避，失败就 `resetLogin` 重新登录。

跑了好几周，稳稳当当。

某天下午，所有任务一起炸。报的是同一个错：`COPY from stdin failed: expected N values, got 1`。Pipeline 重试，再炸，再重试，再炸。循环了好几个小时，没人注意。

重试逻辑在代码里。`resetLogin` 在代码里。

一个也没触发过。

问题出在出事的方式上。ERP 的 session 大概八个半小时过期。过期之后，API 不给你甩 401，不甩 403，不甩任何错误码。它给你 HTTP 200。

Body 拉回来，JSON 一解：

```json
[["error message"]]
```

结构完全合法。一行一列。Go 的 `json.Decode` 没理由报错。代码的判断是 `if err != nil`——err 是 nil，不走进重试，把这个单行单列的结果当成正常数据，接着往数据库里灌。

最后怎么修的？没动重试逻辑。在后台单独起了个 goroutine，每 4 小时主动调一次 login。把 session 续上，不让它过期。

那条重试回路，本来就是为了拦这种故障加的。它从来没真正执行过。

## 二

一个桌面工具，带代码生成功能。用户在可视化编辑器里拖控件、排布局，点一下"生成代码"，工具产出 C 源码给 MCU 用。

输出的文件里，头文件有宏定义，有事件分发，有资源地址索引。看起来齐全。

开发者把这些代码编进固件，烧进芯片。屏幕亮了——所有动态文字控件全挤在 (0, 0)。

编译没 warning。链接全过。没人报错。

排查发现缺了一个文件：控件实例表。这个文件记录每个控件的 id 和它的 x、y、宽、高。MCU 跑起来之后，靠这张表知道每个控件该画在哪。

头文件里声明了控件的结构体，结构体里有 x 和 y 字段。资源的 manifest 里有坐标数据。设计文档里画清楚了控件表的格式。三个地方都提到了坐标这件事。

只有一个问题：代码生成器没把这张表写出来。

那个负责生成这张表的方法，从第一行代码写完就没被调用过。整个管线里没有人意识到它的存在。

不是没审查。审查的人确认了宏定义指向正确的资源地址，确认了事件路由逻辑没问题，确认了页面命名一致。全对。

没有人问：该生成的文件，齐了吗？

## 三

我给一颗新芯片集成 SDK。厂商发了一个 zip 包。

打开看，目录结构都正常。启动文件、链接脚本、芯片型号一一对应。SPI、DMA、GPIO 的驱动都在。

就一个地方有点怪。`startup.S` 的头注释写的是：

```asm
; GCC for CSKY Embedded Processors
```

CSKY 是另一种 CPU 架构。Datasheet 上写得很清楚，这颗芯片是 RISC-V。

但 SDK 目录名标的就是这颗芯片。子目录也对。启动文件好端端待在它该在的位置。注释说 CSKY，目录说 RISC-V——信谁？

最后翻了两个事实出来。

第一，厂商第一次发过来的 SDK 其实是同系列另一颗芯片的。那颗确实是 CSKY。目录名没换过来。

第二，正确的 SDK 到了。打开一看，`startup.S` 头注释还是 CSKY。因为厂商从这个第三方工具链的模板复制过来之后，压根没改过注释。

那实际的指令呢？

```asm
csrw    mtvec, a0
la      sp, g_top_irqstack
jal     main
```

`csrw` 是 RISC-V 写控制状态寄存器的指令，CSKY 没有这个。一行 grep 就验完了：`grep -c "csrw\|mret" startup.S`，返回 3。RISC-V。注释是化石。

## 不是 Agent 傻

三个 Agent。设计、实现、审查。各做了一件事。

单独看，每一件都对。设计写出了和任务一致的 spec。实现写出了和 spec 一致的代码。审查确认了实现和 spec 一致。

每一步验证都通过了。

系统是一致的。一致地错了。

这不是 Agent 的智力不够。Agent 很聪明，你让它查什么它查什么，你划定的圈子里它不会漏。问题出在你没划出来的那块地方。

第一个案例，你让它"检查 HTTP 有没有报错"。它查了，没报错。你没让它"检查成功返回的数据是不是真的数据"。

第二个案例，你让它"检查生成出来的文件对不对"。它查了，都对。你没让它"检查该生成的文件都存在吗"。

第三个案例，你让它"按 SDK 文档配置工程"。它照做了。你没让它"验证一下这个 SDK 的架构跟你 datasheet 写的是不是一回事"。

三次是一样的。Agent 在它知道的范围里做了完整的验证。范围本身是错的。

这不是 bug。是 workflow 缺了一层。

Agent 不会自己琢磨"还有什么该查的我没查"。那个问题——"我查了的都对，但有什么是我根本不知道要查的"——不能从 Agent 自己的上下文里长出来。它只能从外面来。

---

## 那该补什么

我往自己的开发流程里加了三条验证。都是很小的东西，放在任务描述或者审查步骤后面就行。

**一、语义验证。**

不只是看函数有没有报错。看一下报错的方式。

HTTP 200 拿到手，别急着往下走。扫一眼结构：行数对吗？列数对吗？有没有"error""失败""错误"这种关键词混在数据里？如果你请求了几十列却只拿回来一行，这个结果不管 err 是什么，已经不对了。

API 的"成功"只是传输层成功了。你不验语义，等于没验。

**二、产物完整性。**

代码生成器产出一个目录。别只看里面的文件内容对不对。先点一遍：该有的都在吗？

生成之前列个清单，生成完了对着清单一个一个确认。不是"已有的文件没问题"，是"全部文件都存在且不空"。这两句话差很远。

**三、信息源验证。**

凡是可以用命令行验证的东西，就不要信人写的说明。

注释会说谎。文件名会说谎。目录结构会说谎。一个 `.S` 文件是哪家的 CPU 架构，注释说了不算，指令说了算。一个二进制文件的格式，README 说了不算，特定偏移位置上的字节值说了算。

在 Agent 的流程里，每当你准备相信一个技术前提往下走之前，停半秒。能不能找个命令验证它？能就验。别把别人的"声明"当成你的"事实"传给下一步。

---

三条规则都很小。不用动流程结构，就是多问一句的事。

但合在一起，刚好拦住我反复踩的三类坑：返回成功但数据是错的、生成了代码但缺了文件、以及工具链里循环传递的假标签。

Agent 不会问你"还该验证什么"。那个问题只能你想。你搭多 Agent 工作流，不是光在设计一条产代码的流水线。你是在设计每个 Agent 信什么、用什么当真的。

你在 Agent 的边界上放验证点了，就不会漏。不放，没人替你放。
