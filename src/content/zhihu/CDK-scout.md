十一项测试全部通过。agent 说测试覆盖够了，我说 commit，push。我们把一段 C 代码推了上去——问题是，这段代码从来没有经过真正的编译。

我做后端和分布式系统，不写固件。这个项目是一个桌面工具，负责给 MCU 生成 C 代码，生成出来的代码最终要进入 CDK（芯片厂商提供的 IDE）编译，目标是 RISC-V 架构，跑在 Windows 上，还嵌在一套我完全陌生的 package system 里。而我开发环境是 Linux。那十一项测试，其实只是用本机 gcc 跑的——给本机编译程序的普通编译器。真正上线需要的是另一套编译器、另一套目标架构、另一套构建环境。两台编译器，两个世界。我的测试，全都跑在错误的世界里。

一天后，Windows 那边传回来第一条报错。然后第二条。第五条。接着第七条。

---

## 七轮调试

每次报错回来，agent 的反应几乎完全一致：先给一句非常确定的判断——"根因是……"——然后提交一个 fix，push，等待下一轮。第一轮，头文件漏了拷贝。第二轮，类型名写错，这种错误 host gcc 根本碰不到。第三轮，SDK 压根没参与编译，因为 manifest 里少了一项冷门配置。第四轮，同一个 GPIO 引脚，在 agent 参考的 SDK 里叫一个名字，在 CDK 实际带的 SDK 里叫另一个名字。第五轮，一个完整驱动文件被编译成空文件，因为控制它的开关藏在生成文件里，编译器根本看不到。第六轮，上一枪只修了表面。第七轮，才发现真正的问题在架构。

七轮。每一轮 agent 都说自己找到了"根因"。每一轮都确实修复了真实问题。但每一轮，都只是下一层问题的外壳。

真正让我在意的不是这些 bug，而是整个循环的形状。agent 三秒钟给出答案，验证它却要花我几分钟到几小时——Windows 上拉代码、跑完整 CDK 编译、读报错、把 stderr 复制回来。我搭了一套复杂的 multi-agent workflow，整个系统里唯一真正接触现实的组件，是一个负责复制粘贴的人。也就是我。

---

## 为什么它永远停不下来

最危险的地方不是 agent 会犯错，而是它永远用同样的语气犯错。每一轮它都说"根因找到了"，但这种确定性根本不代表它正在接近答案——它可能在逼近，也可能只是随机试错。有一轮它甚至建议我直接删掉那个导致驱动文件为空的编译开关，被我一句"改动太脏"否掉了。后来真正的解法完全不同：不是继续修补编译开关，而是把选择驱动的逻辑从编译期移到链接期——编译哪份 `.c` 文件，就激活哪个驱动。架构修正，不是补丁。

而整个过程中，agent 从来没有说过一句最该说的话：**"我看不到真实目标环境。后面所有 fix，本质上都只是猜测。别继续 pull 了——先把 RISC-V 工具链装到本地，否则我一直在盲猜。"** 它不会这么说，因为每猜一次花它三秒，花我一个下午，而它不付这笔账。一个不承担试错成本的系统，天然倾向于继续猜，而不是先获取更多信息。经济性是沉默的，完全落在键盘前那个人身上。

---

## 提前解释，并不能解决问题

最直观的教训似乎是：我应该一开始就把 CDK 的规则讲清楚——manifest 怎么写，组件怎么拉进构建系统，文件放哪个位置。但后来我发现，即使提前解释，也解决不了核心问题。

因为 agent 其实并不缺 CDK 知识。它懂得相当多——能从零搭组件、写 manifest、配 search path。甚至有一轮，它长篇大论解释了 CDK 如何解析组件依赖，逻辑严密，听起来完全正确。问题是那套解释是错的。AI 最危险的地方从来不是无知，而是半懂——它知道系统的大致形状，但不知道那个特定版本、接上那套特定 SDK、构建跑起来那一刻的真实行为。漏掉的配置项、manifest 的格式差异、SDK 版本漂移带来的引脚名变化——这些都不是"知识问题"，而是只有对着目标真编译才会暴露的运行时现实。它知道的和它看不透的紧紧挨着，中间没有一道缝。

一个有经验的嵌入式工程师，看到"同一颗芯片，两套 SDK"，警报立刻就会响。被版本坑过太多次，本身触发条件反射：同 chip、两套 SDK，符号表必然对不上，先 diff。这不是什么超能力——给对 prompt，模型也能拉出同样的排查清单：查符号差异、diff 引脚定义、对比寄存器基地址。差别不在能力，在默认姿态。工程师的经验沉淀把默认值重写成了怀疑；模型的默认值是自信，它会冲。有人推一把它能切成防御，但那次没人推，因为我们谁都不知道有坑。而第二个理由更直白，跟模型多聪明没关系——Windows上的 SDK 根本不在它的上下文里。agent 手里自始至终只有一份。

---

## 探路者

我刚开始真正该做的事不是写代码，而是先把 RISC-V 工具链装到本地——哪怕不生成真实二进制，哪怕只是跑一次语法检查。因为这样至少能把目标环境先拉近一点，让符号差异在本地秒级暴露，而不是靠Copy-Paste传回来。

这改变不了 trial-and-error 这种试错方式。面对真正未知的问题，试错是唯一的探索方式，不是所有错误都能靠"想得足够多"预知。你能做的只是把试错成本降到最低——先用最小的代价把未知变成已知。

---

## 我后来做的三件事

项目终于跑通之后,我回头改了整个流程。三件事，值得诚实说清楚。

**第一件：打通流程。** 这是最有效的一步。代码生成器以前只生成源码，CDK 工程文件需要手工配置——七轮里有两轮问题都来自人手配置漏了项。我直接改了生成器，让它把 CDK 工程需要的所有配置一并生成——源文件列表、头文件路径、内存布局、构建参数，整个手工步骤不复存在。我以前总想着加检查。这次才明白：与其反复校验，不如让需要校验的东西直接消失。从源头删掉一个出错的可能，比事后检查多少次都管用。

**第二件：把检查拉近一步，然后承认它的局限。** 我加了一步本地语法检查，生成代码之后直接跑编译分析。类型错误、缺失声明、头文件问题，从"几天后的剪贴板"拉到了"秒级本地"。但它跑在 host gcc 下——我本机的编译器，不是 RISC-V。所以 SDK 版本差异、manifest 的问题、那个编译器看不到的开关——没一个它能抓，因为它们活在 host gcc 永远进不去的世界里。检查拉近了，但它还是没覆盖真正会出问题的地方。要补上这块——在本地每次构建时接入 RISC-V 工具链跑同一条语法检查——我还没做。我知道答案，它就在脚边地上。这是这个系统诚实的现状。

**第三件：把踩过的坑记下来。** 我把那七轮试出来的经验写成了规则 —— manifest 格式要求、那个容易遗漏的配置项、一张 SDK 版本间引脚名差异的表。沉淀了七轮试错换来的知识。这一步确实会生效—— AGENTS.md 在每个 session 启动时都会被加载。

## harness 到底是什么

以前我一直以为 AI coding 最重要的那步是验证——验行为、验规格、搜同类 bug。这都对，我前面写过。这次教了我那一步底下那一步。

agent 只在它的上下文里思考。这不是你能修复的局限。它懂那套构建环境的大致样子——懂到听起来、做起来都真实地胜任。但它没法知道的是，这个特定目标在构建真正跑起来那一刻会怎么表现，它感觉不到两者之间的接缝。它知道的和它会踩进去的，紧紧挨着，中间没有一道缝。所以它没说"我不知道"，古圣人所说的 - 不知而自以为知。 

所有要有一个人站在那儿，不停扩展Agent 的上下文边界。这个人不需要理解边界外面是什么 —— 我不理解。这就是 harness。不是单元测试/集成测试。harness 的本质，是缩短 agent 和真实世界之间的距离。

我不会写 firmware 。直到今天，我依然不懂那颗 RISC-V 芯片。但我是整个系统里唯一能让它真正上线的人，不是因为我懂嵌入式，而是因为——当 agent 写的代码多到你看不完，人剩下的工作就只剩一件事：识别Agent的能力边界，并不断扩展。未来工程师的稀缺价值，越来越不在于写代码，而在于知道 Agent 不知道的领域。
